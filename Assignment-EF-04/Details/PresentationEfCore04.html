Loading Related Data

Section 01

Loading Related Data 📦

When you have relationships (navigation properties), you need to decide how and when to load the related data. EF Core offers three strategies.

Strategy When Loaded How Queries
Eager With main query .Include() 1 (with JOIN)
Explicit On demand .Entry().Load() Separate query
Lazy When accessed Auto (proxies) Per navigation ⚠️

The Problem: Related Data is NULL by Default

var product = context.Products.First();
Console.WriteLine(product.Category.Name);  // ❌ NullReferenceException!
// Category was NOT loaded!

👆 Click to reveal

Eager Loading — Include() (Single Level)

Eager Loading loads related data in the same query using JOINs. Use Include() for direct (single-level) relationships.

LINQ Query

// Load Product WITH its Category
var products = context.Products
    .Include(p => p.Category)  // ← Load related Category
    .ToList();

// Now this works!
Console.WriteLine(products[0].Category.Name);  // ✅ "Electronics"

Generated SQL

SELECT [p].[Id], [p].[Name], [p].[Price], [p].[CategoryId],
       [c].[Id], [c].[Name]
FROM [Products] AS [p]
LEFT JOIN [Categories] AS [c] 
    ON [p].[CategoryId] = [c].[Id]

Multiple Includes (Same Level)

// Load Order with Customer AND OrderItems
var orders = context.Orders
    .Include(o => o.Customer)      // First relationship
    .Include(o => o.OrderItems)    // Second relationship
    .ToList();
-- Generated SQL (two LEFT JOINs)
SELECT [o].[Id], [o].[OrderDate], [o].[CustomerId],
       [c].[Id], [c].[Name], [c].[Email],
       [oi].[Id], [oi].[OrderId], [oi].[ProductId], [oi].[Quantity]
FROM [Orders] AS [o]
LEFT JOIN [Customers] AS [c] ON [o].[CustomerId] = [c].[Id]
LEFT JOIN [OrderItems] AS [oi] ON [o].[Id] = [oi].[OrderId]

👆 Click to reveal

Eager Loading — ThenInclude() (2 Levels)

Use ThenInclude() to load nested relationships — relationships of relationships. Chain multiple levels deep.

LINQ Query (2 Levels Deep)

// Load Order → OrderItems → Product
var orders = context.Orders
    .Include(o => o.OrderItems)          // Level 1: Order's Items
        .ThenInclude(oi => oi.Product)    // Level 2: Each Item's Product
    .ToList();
-- Generated SQL
SELECT [o].[Id], [o].[OrderDate], [o].[CustomerId],
       [oi].[Id], [oi].[OrderId], [oi].[ProductId], [oi].[Quantity],
       [p].[Id], [p].[Name], [p].[Price], [p].[CategoryId]
FROM [Orders] AS [o]
LEFT JOIN [OrderItems] AS [oi] ON [o].[Id] = [oi].[OrderId]
LEFT JOIN [Products] AS [p] ON [oi].[ProductId] = [p].[Id]

👆 Click to reveal

Eager Loading — ThenInclude() (3+ Levels)

LINQ Query (3+ Levels Deep)

// Load Order → Customer
//           → OrderItems → Product → Category
var orders = context.Orders
    .Include(o => o.Customer)                     // Branch 1
    .Include(o => o.OrderItems)                   // Branch 2: Level 1
        .ThenInclude(oi => oi.Product)             // Branch 2: Level 2
            .ThenInclude(p => p.Category)           // Branch 2: Level 3
    .ToList();
-- Generated SQL (multiple JOINs)
SELECT [o].[Id], [o].[OrderDate], [o].[CustomerId],
       [c].[Id], [c].[Name], [c].[Email],
       [oi].[Id], [oi].[OrderId], [oi].[ProductId], [oi].[Quantity],
       [p].[Id], [p].[Name], [p].[Price], [p].[CategoryId],
       [cat].[Id], [cat].[Name]
FROM [Orders] AS [o]
LEFT JOIN [Customers] AS [c] ON [o].[CustomerId] = [c].[Id]
LEFT JOIN [OrderItems] AS [oi] ON [o].[Id] = [oi].[OrderId]
LEFT JOIN [Products] AS [p] ON [oi].[ProductId] = [p].[Id]
LEFT JOIN [Categories] AS [cat] ON [p].[CategoryId] = [cat].[Id]

⚠️ Performance Warning: Deep includes with many JOINs can cause "cartesian explosion" — loading duplicate data. Consider AsSplitQuery() for complex graphs!

Eager Loading — Filtered Include

Filtered Include (EF Core 5+) lets you filter and order the related data being loaded. This is powerful for reducing data transfer!

LINQ Query

// Load Category with only active, expensive products
var categories = context.Categories
    .Include(c => c.Products
        .Where(p => p.IsActive && p.Price > 100)  // Filter!
        .OrderBy(p => p.Name)                      // Order!
        .Take(5))                                  // Limit!
    .ToList();

// Only loads 5 active products > $100 per category

👆 Click to reveal

Filtered Include — Supported Operations

Generated SQL

SELECT [c].[Id], [c].[Name], [t].[Id], [t].[Name], [t].[Price], [t].[IsActive]
FROM [Categories] AS [c]
LEFT JOIN (
    SELECT [p].[Id], [p].[Name], [p].[Price], [p].[IsActive], [p].[CategoryId]
    FROM [Products] AS [p]
    WHERE [p].[IsActive] = 1 AND [p].[Price] > 100.0  -- Filter applied!
    ORDER BY [p].[Name]                                -- Order applied!
    OFFSET 0 ROWS FETCH NEXT 5 ROWS ONLY             -- Take(5)!
) AS [t] ON [c].[Id] = [t].[CategoryId]
ORDER BY [c].[Id], [t].[Name]

Supported Operations in Filtered Include

Operation Supported Example
Where() ✅ Yes Filter the collection
OrderBy() / OrderByDescending() ✅ Yes Sort related data
Take() / Skip() ✅ Yes Limit/paginate
AsNoTracking() ✅ Yes Disable change tracking
Select() ❌ No Cannot project within Include
Include() ❌ No Cannot use Include() inside the filter lambda

AsSplitQuery() — Avoiding Cartesian Explosion

When you use multiple .Include() on collections, EF Core generates a single query with multiple JOINs. This can cause cartesian explosion — duplicated data that slows everything down.

The Problem — Single Query (Default)

// Default: One big query with JOINs → cartesian explosion!
var orders = context.Orders
    .Include(o => o.OrderItems)  // 10 items
    .Include(o => o.Payments)    // 3 payments
    .ToList();
// Result: 10 × 3 = 30 rows returned! (lots of duplicate data)

The Solution — AsSplitQuery() (EF Core 5+)

// ✅ Split into separate, efficient queries
var orders = context.Orders
    .Include(o => o.OrderItems)
    .Include(o => o.Payments)
    .AsSplitQuery()  // ← 3 separate queries, no explosion
    .ToList();
// Query 1: SELECT * FROM Orders
// Query 2: SELECT * FROM OrderItems WHERE OrderId IN (...)
// Query 3: SELECT * FROM Payments WHERE OrderId IN (...)

When to Use Each

ApproachBest ForTrade-off
Single Query (default)Simple includes, 1-2 relationshipsMay duplicate data
AsSplitQuery()Multiple collection includesMultiple DB round-trips

Explicit Loading — Reference (Single Entity)

Explicit Loading loads related data after the main entity is already loaded. Use Reference() for single navigation properties (one-to-one or many-to-one).

LINQ Query

// Step 1: Load product without related data
var product = context.Products.First(p => p.Id == 1);

// Step 2: Load Category explicitly (separate query)
context.Entry(product)
    .Reference(p => p.Category)
    .Load();

Console.WriteLine(product.Category.Name);  // ✅ Now available!

Generated SQL (Two Separate Queries)

-- Query 1: Load product
SELECT TOP(1) [p].[Id], [p].[Name], [p].[Price], [p].[CategoryId]
FROM [Products] AS [p]
WHERE [p].[Id] = 1

-- Query 2: Load category (triggered by .Load())
SELECT [c].[Id], [c].[Name]
FROM [Categories] AS [c]
WHERE [c].[Id] = @categoryId  -- Uses product.CategoryId

Check If Already Loaded

// Avoid loading if already loaded
var entry = context.Entry(product);

if (!entry.Reference(p => p.Category).IsLoaded)
{
    entry.Reference(p => p.Category).Load();
}

// IsLoaded = true after Load() is called

👆 Click to reveal

Explicit Loading — Collection (Basic)

Use Collection() for collection navigation properties (one-to-many). You can also use Query() to filter, sort, or paginate the loaded data!

LINQ Query (Basic)

// Step 1: Load category without products
var category = context.Categories.First(c => c.Id == 1);

// Step 2: Load Products collection explicitly
context.Entry(category)
    .Collection(c => c.Products)
    .Load();

Console.WriteLine(category.Products.Count);  // ✅ Now available!

Explicit Loading — Filtered Collection with Query()

Generated SQL

-- Query 1: Load category
SELECT TOP(1) [c].[Id], [c].[Name]
FROM [Categories] AS [c]
WHERE [c].[Id] = 1

-- Query 2: Load products (triggered by .Load())
SELECT [p].[Id], [p].[Name], [p].[Price], [p].[CategoryId]
FROM [Products] AS [p]
WHERE [p].[CategoryId] = 1  -- Loads ALL products in category

Filtered Collection with Query()

// Load only expensive, active products
context.Entry(category)
    .Collection(c => c.Products)
    .Query()                              // Returns IQueryable!
    .Where(p => p.Price > 100 && p.IsActive)
    .OrderBy(p => p.Name)
    .Take(10)
    .Load();
-- Generated SQL (filtered)
SELECT [p].[Id], [p].[Name], [p].[Price], [p].[CategoryId]
FROM [Products] AS [p]
WHERE [p].[CategoryId] = 1 
  AND [p].[Price] > 100.0 
  AND [p].[IsActive] = 1
ORDER BY [p].[Name]
OFFSET 0 ROWS FETCH NEXT 10 ROWS ONLY

💡 Pro Tip: Use .Query() to get an IQueryable and apply any LINQ operations — great for loading subset of large collections!

Lazy Loading

Lazy Loading automatically loads related data when you access a navigation property. Sounds convenient, but it's often a performance trap!

Enabling Lazy Loading

// Step 1: Install package
// dotnet add package Microsoft.EntityFrameworkCore.Proxies
 
// Step 2: Enable in DbContext
optionsBuilder.UseLazyLoadingProxies();
 
// Step 3: Make navigation properties virtual
public class Product
{
    public int Id { get; set; }
    public virtual Category Category { get; set; }  // virtual!
}

⚠️ The N+1 Problem: Lazy loading in a loop causes N+1 queries!

👆 Click to reveal

The N+1 Problem — Why Lazy Loading is Dangerous

The N+1 problem happens when you access navigation properties inside a loop. Instead of 1 query, EF Core sends N+1 queries to the database!

The Dangerous Code

// This looks innocent... but it's a TRAP! 🪤
var categories = context.Categories.ToList(); // Query 1

foreach (var category in categories) // 10 categories
{
    // ❌ Each access triggers a NEW query!
    Console.WriteLine($"Category: {category.Name}");
    Console.WriteLine($"Products: {category.Products.Count}"); // Query 2, 3... 11!
}
// Total: 1 + 10 = 11 queries!

The Fix — Use Eager Loading Instead!

// ✅ One query loads everything with a JOIN
var categories = context.Categories
    .Include(c => c.Products)
    .ToList(); // Only 1 query!

foreach (var category in categories)
{
    Console.WriteLine($"Category: {category.Name}");
    Console.WriteLine($"Products: {category.Products.Count}"); // ✅ Already loaded!
}
⚠️ Real-World Impact: With 1000 categories, N+1 sends 1001 queries instead of 1. This is the #1 performance killer in EF Core applications!

🤔 Quick Think — Loading Related Data

Q1: Product listing page shows each product with its category name. Which loading strategy?

Answer: Eager Loading with .Include(p => p.Category)

You know upfront you need the category for every product. One query with a JOIN is the most efficient approach!

Q2: A user clicks a product and SOMETIMES wants to see reviews. Which loading strategy?

Answer: Explicit Loading

Load reviews on demand only when needed:
context.Entry(product).Collection(p => p.Reviews).Load();

Q3: What is the N+1 problem and why is it dangerous?

Answer: Accessing navigation properties in a loop with Lazy Loading sends N+1 queries (1 for main entity + N for each related entity).

With 1000 records = 1001 DB queries! Always prefer Eager Loading for known relationships.

Q4: When would you use AsSplitQuery()?

Answer: When you have multiple .Include() calls on collection navigation properties.

It prevents cartesian explosion (data duplication from multiple JOINs) by splitting into separate queries.

Inheritance Strategies

Section 02

Inheritance Strategies 🏗️

EF Core supports C# inheritance hierarchies. There are two main strategies for mapping inheritance to database tables: TPH and TPT.

Strategy Full Name Tables Best For
TPH Table-per-Hierarchy 1 table for all types Performance (default)
TPT Table-per-Type Separate table per type Data integrity
TPC Table-per-Concrete Table per concrete class Rare use cases (EF7+)
💡 Recommendation: Start with TPH (default) — it's the most performant for most scenarios. Only switch to TPT or TPC if you have specific data integrity or schema requirements.

👆 Click to reveal

TPH — Table-per-Hierarchy (Default)

TPH stores the entire inheritance hierarchy in a single table. A discriminator column indicates which type each row represents.

TPH Table Structure

-- Single "Payments" table
-- ┌────┬──────────┬────────────┬─────────────┬──────────────┬─────────────┐
-- │ Id │ Amount   │ Date       │Discriminator│ CardNumber   │ BankAccount │
-- ├────┼──────────┼────────────┼─────────────┼──────────────┼─────────────┤
-- │ 1  │ 100.00   │ 2024-01-01 │ CreditCard  │ 4111...      │ NULL        │
-- │ 2  │ 250.00   │ 2024-01-02 │ BankTransfer│ NULL         │ EG12345...  │
-- └────┴──────────┴────────────┴─────────────┴──────────────┴─────────────┘

Fluent Configuration

// Customize discriminator
modelBuilder.Entity<Payment>()
    .HasDiscriminator<string>("PaymentType")
    .HasValue<CreditCardPayment>("CreditCard")
    .HasValue<BankTransferPayment>("BankTransfer");

✅ TPH Pros: Fast queries (no JOINs), simple
⚠️ TPH Cons: Many NULL columns, can't enforce NOT NULL on derived types

👆 Click to reveal

TPT — Table-per-Type

TPT creates a separate table for each type in the hierarchy. Base class properties go in the base table; derived properties go in derived tables.

TPT Table Structure

-- "Payments" table (base)
-- ┌────┬──────────┬────────────┐
-- │ Id │ Amount   │ Date       │
-- └────┴──────────┴────────────┘

-- "CreditCardPayments" table (derived)
-- ┌────┬──────────────┐
-- │ Id │ CardNumber   │   (FK to Payments.Id)
-- └────┴──────────────┘

-- "BankTransferPayments" table (derived)
-- ┌────┬─────────────┐
-- │ Id │ BankAccount │   (FK to Payments.Id)
-- └────┴─────────────┘

Fluent Configuration

// Map each type to its own table
modelBuilder.Entity<Payment>().ToTable("Payments");
modelBuilder.Entity<CreditCardPayment>().ToTable("CreditCardPayments");
modelBuilder.Entity<BankTransferPayment>().ToTable("BankTransferPayments");

✅ TPT Pros: No NULL columns, better data integrity
⚠️ TPT Cons: Requires JOINs (slower queries)

👆 Click to reveal

TPC — Table Structure (EF Core 7+)

TPC (Table Per Concrete Type) creates a separate table for each concrete (non-abstract) type. Each table contains ALL columns including inherited ones. No base table is created for abstract classes.

Generated Tables

┌─────────────────────────────────────────────────────────────────────────────┐
│                    TPC: TABLE PER CONCRETE TYPE                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ❌ NO Animals table (abstract class)                                      │
│                                                                             │
│   ┌─────────────────────────────┐    ┌─────────────────────────────┐       │
│   │         Cats Table          │    │         Dogs Table          │       │
│   ├─────────────────────────────┤    ├─────────────────────────────┤       │
│   │ Id (PK)        INT          │    │ Id (PK)        INT          │       │
│   │ Name           NVARCHAR     │    │ Name           NVARCHAR     │       │
│   │ Age            INT          │    │ Age            INT          │       │
│   │ MeowSound      NVARCHAR     │    │ BarkSound      NVARCHAR     │       │
│   └─────────────────────────────┘    └─────────────────────────────┘       │
│         ▲                                   ▲                               │
│         │                                   │                               │
│    ALL columns including              ALL columns including                │
│    inherited (Id, Name, Age)          inherited (Id, Name, Age)            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

TPC — Configuration & Key Generation 🔑

Key Generation Challenge

⚠️ Important: With TPC, you need unique IDs across ALL tables! Use Sequences or GUIDs to avoid duplicate IDs. 

// Use a shared sequence for unique IDs
modelBuilder.HasSequence<int>("AnimalSequence");

modelBuilder.Entity<Cat>()
    .Property(c => c.Id)
    .HasDefaultValueSql("NEXT VALUE FOR AnimalSequence");

modelBuilder.Entity<Dog>()
    .Property(d => d.Id)
    .HasDefaultValueSql("NEXT VALUE FOR AnimalSequence");

Fluent API Configuration

protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    // Option 1: Use TPC mapping strategy (EF Core 7+)
    modelBuilder.Entity<Animal>().UseTpcMappingStrategy();

    // Option 2: Or map each concrete type to its own table
    modelBuilder.Entity<Cat>().ToTable("Cats");
    modelBuilder.Entity<Dog>().ToTable("Dogs");
}

🤔 Quick Think

Q1: What are the three inheritance mapping strategies in EF Core?

Answer:
1. TPH (Table Per Hierarchy) — All types in ONE table + discriminator column (default)
2. TPT (Table Per Type) — Separate table for each type, joined by PK
3. TPC (Table Per Concrete Type) — Separate table for each concrete type, no base table

TPH is the default and most performant for most scenarios.

Q2: What is a discriminator column in TPH?

Answer: A discriminator is a column that stores the type name to identify which derived type each row represents.

Default name: Discriminator
Values: Type names (e.g., "Blog", "RssBlog")

Customize:
.HasDiscriminator<string>("BlogType")
.HasValue<Blog>("blog_base")
.HasValue<RssBlog>("blog_rss")

Q3: When would you choose TPT over TPH?

Answer: Choose TPT when:

• Derived types have many unique columns (TPH would have lots of NULLs)
• You need database-level constraints on derived type columns
• Normalized schema is a requirement

Trade-off: TPT requires JOINs for queries, which can be slower than TPH.

Configure: .ToTable("RssBlogs") on each derived type

Q4: How do you configure TPC (Table Per Concrete Type)?

Answer: Use UseTpcMappingStrategy() (EF Core 7+):

modelBuilder.Entity<Blog>().UseTpcMappingStrategy();

Or map each concrete type to its own table:
modelBuilder.Entity<RssBlog>().ToTable("RssBlogs");
modelBuilder.Entity<PhotoBlog>().ToTable("PhotoBlogs");

No base table is created. Each table has ALL columns including inherited ones.

LINQ Joins

Section 03

LINQ — Join Category 🔗

Joins in LINQ allow you to combine data from two or more sequences based on a matching key — just like SQL JOINs, but written in pure C# with full type safety and IntelliSense support.

Join Types We'll Cover

  • 🔵 Inner Join — Matching elements from both sequences
  • 📦 Group Join — One-to-many, like a SQL LEFT JOIN with grouping
  • ⬅️ Left Outer Join — All left elements, even without a match
  • ✖️ Cross Join — Every combination of both sequences
  • 🆕 LeftJoin / RightJoin — First-class operators in EF Core 10+ (no more GroupJoin gymnastics!)

💡 Key Insight: LINQ joins work on IEnumerable<T> (in-memory) and IQueryable<T> (EF Core → SQL). 🆕 EF Core 10 adds LeftJoin() and RightJoin() — making outer joins much simpler!

👆 Click to reveal

Inner Join — Matching Both Sides 🔵

An Inner Join returns only the elements where the key exists in both sequences. Non-matching elements are excluded from the result — same behavior as SQL INNER JOIN.

Query Syntax

var result = from order in orders
             join customer in customers
             on order.CustomerId equals customer.Id
             select new
             {
                 customer.Name,
                 order.Product,
                 order.Total
             };

Method Syntax

var result = orders.Join(
    customers,
    order    => order.CustomerId,   // outer key
    customer => customer.Id,        // inner key
    (order, customer) => new
    {
        customer.Name,
        order.Product,
        order.Total
    });

⚠️ Watch Out: Both keys must use equals (not ==) in query syntax. Using == will cause a compile error!

👆 Click to reveal

Group Join — One-to-Many Grouping 📦

A Group Join correlates each element of the outer sequence with a collection of matching inner elements. It's the backbone of Left Outer Joins and hierarchical projections like "customer with all their orders".

Query Syntax

var result = from customer in customers
             join order in orders
             on customer.Id equals order.CustomerId
             into customerOrders          // group into collection
             select new
             {
                 customer.Name,
                 Orders = customerOrders  // IEnumerable of matching orders
             };

Method Syntax

var result = customers.GroupJoin(
    orders,
    customer => customer.Id,
    order    => order.CustomerId,
    (customer, customerOrders) => new
    {
        customer.Name,
        Orders = customerOrders
    });

💡 Pro Tip: If a customer has no orders, customerOrders is an empty collection — not null. That's what makes it a safe Left Join foundation!

👆 Click to reveal

Left Outer Join — .LeftJoin() (EF Core 10+) ⬅️

.NET 10 finally adds a first-class LeftJoin() operator! No more GroupJoin + SelectMany + DefaultIfEmpty gymnastics.

🆕 EF Core 10 — Clean & Simple!

// ✅ EF Core 10: First-class LeftJoin operator
var result = customers.LeftJoin(
    orders,
    customer => customer.Id,
    order => order.CustomerId,
    (customer, order) => new
    {
        customer.Name,
        Product = order?.Product ?? "No Orders",
        Total = o != null ? o.Number : 0
    });
// Generated SQL: LEFT JOIN — identical to hand-written SQL!
💡 .NET 10 also adds RightJoin() — keeps all right elements instead. Only method syntax supported — no query syntax keyword yet!
⚠️ Handling nulls: The right side can be null when there's no match. Always use ?. and ?? to avoid NullReferenceException!

👆 Click to reveal

Left Outer Join — Classic Pattern (Pre-.NET 10) ⬅️

Before .NET 10, LINQ had no dedicated left join operator. You had to combine GroupJoin + SelectMany + DefaultIfEmpty:

Query Syntax (Classic)

var result = from customer in customers
             join order in orders
             on customer.Id equals order.CustomerId
             into customerOrders
             from order in customerOrders.DefaultIfEmpty() // ← the magic
             select new
             {
                 customer.Name,
                 Product = order?.Product ?? "No Orders",
                 Total = o != null ? o.Number : 0
             };

Method Syntax (Classic)

var result = customers
    .GroupJoin(orders,
        c => c.Id,
        o => o.CustomerId,
        (c, ords) => new { c, ords })
    .SelectMany(
        x => x.ords.DefaultIfEmpty(),
        (x, order) => new
        {
            x.c.Name,
            Product = order != null ? order.Product : "No Orders"
        });
⚠️ Legacy Pattern: DefaultIfEmpty() returns null for missing right elements — always use null-conditional operators (?. and ??) to avoid NullReferenceException!

👆 Click to reveal

Cross Join — Every Combination ✖️

A Cross Join (Cartesian Product) pairs every element of the first sequence with every element of the second. If sequence A has 3 items and B has 4, you get 12 results. No key matching is involved.

Query Syntax

// Just use two "from" clauses — no "join" keyword needed!
var result = from size  in sizes
             from color in colors
             select new
             {
                 Size  = size,
                 Color = color
             };
// sizes  = ["S","M","L"]  |  colors = ["Red","Blue"]
// → 6 results: S-Red, S-Blue, M-Red, M-Blue, L-Red, L-Blue

Method Syntax

var result = sizes.SelectMany(
    size  => colors,
    (size, color) => new { Size = size, Color = color }
);

⚠️ Performance Warning: Cross joins grow exponentially! 1000 × 1000 = 1,000,000 results. Always question whether a cross join is truly what you need.

🤔 Quick Think — LINQ Joins

Q1: You have a list of students and enrollments. You want only students who ARE enrolled. Which join?

Answer: Inner Join

Returns only matching records from both sides — students without enrollments are excluded.

Q2: You want ALL students, showing their enrollments if any (or "Not Enrolled"). Which join?

Answer: Left Outer Join

All left elements, with null for missing right elements.
Pattern: GroupJoin + SelectMany + DefaultIfEmpty

Q3: When working with EF Core, should you prefer manual LINQ joins or Include()?

Answer: Prefer Include() and ThenInclude() for navigation properties — EF Core generates optimized SQL for you.

Use manual joins only when navigation properties aren't available or for complex custom projections.
